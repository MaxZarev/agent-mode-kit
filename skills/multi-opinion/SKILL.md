# multi-opinion — совет моделей по открытой проблеме

Созывает несколько внешних AI-CLI (**codex + gemini + claude + glm**)
параллельно как **совет**: каждый независимо изучает проблему и код (read-only)
и предлагает, **как её решить**. Ведущий агент затем сводит предложения в одну
рекомендацию, проверив их по реальному коду.

Ценность — **разнообразие подходов и слепых зон**: четыре модели предложат
варианты, которые один агент бы не увидел. Риск — **галлюцинированные или
неприменимые к этому коду идеи**; ведущий агент обязан их отсеять.

## Чем отличается от `multi-review`

- `multi-review` — **критика готового артефакта** (diff, спека, план): «что тут
  не так». Смотрит назад.
- `multi-opinion` — **генерация решений открытой проблемы**: «как это сделать».
  Смотрит вперёд. Никто ничего не правит — только предлагают.

Если пользователь хочет проверить уже сделанное — это `multi-review`. Если
застрял и хочет идеи/варианты — это `multi-opinion`.

## Когда использовать

- Основной агент застрял: перепробовал несколько вариантов, ничего не работает.
- Архитектурная развилка: несколько способов сделать, неясно какой лучше.
- Непонятный баг: нужны независимые гипотезы о причине и способах починить.
- Пользователь просит «совет моделей» / «спроси несколько ИИ как решить».

НЕ используй, когда задача тривиальна, когда нужен ровно один советник (вызови
его CLI напрямую — `/codex`, `/glm`), или когда речь о ревью готового изменения
(это `multi-review`).

## Рабочий процесс

1. **Собери контекст проблемы.** Ведущий агент пишет файл (например
   `/tmp/mo-problem.md`), куда честно кладёт:
   - что именно решаем (цель), и в каком месте кода это живёт;
   - какие варианты уже пробовали и **почему они не сработали**;
   - что сейчас работает, а что нет; ограничения и нефункциональные требования;
   - какие файлы/функции наиболее релевантны (чтобы советники не искали вслепую).

   Чем честнее «что пробовали / что не работает» — тем меньше совет предложит
   уже отвергнутого.

2. **Собери пакет.** Запусти
   `scripts/build-opinion-pack.sh <context_root> /tmp/mo-problem.md`
   (см. `references/cli-commands.md`). `context_root` — каталог с кодом для
   read-only инспекции (или `-`, если кода нет). Билдер пишет
   `/tmp/multi-opinion/prompt.md` и редактирует шаблоны секретов.

3. **Проверь доступные CLI.** Блок проверки из `references/cli-commands.md`.
   Нужен хотя бы один из codex / gemini / claude / glm.

4. **Запусти советников параллельно.** Один **отдельный** вызов
   `scripts/run-advisor.sh` на каждого, `run_in_background: true`,
   `timeout: 900000`. Не объединяй через `&&`. Codex с gpt-5.5/xhigh может идти
   3–10+ минут — не блокируйся, дождись фоновых уведомлений без `sleep`-поллинга.

5. **Полные находки — в файл (ДО синтеза).** Прочитай `<advisor>.out` каждого
   советника и выпиши в один читаемый md-файл **всё содержательное**: подход,
   шаги, аргументы, риски, на какие файлы/код ссылался — полный разбор по
   советникам, а не пересказ в две строки. Сырой лог CLI (рассуждения,
   tool-calls, вода) в файл не копируй. Собери файл в
   `/tmp/multi-opinion/findings.md`, затем **перенеси в рабочую папку текущей
   задачи** (туда, где живут её рабочие файлы: для подготовки урока — папка
   сессии; иначе — корень проекта) под именем `council-<YYYY-MM-DD>-<тема>.md`
   и покажи путь пользователю.

6. **Синтезируй.** По `references/synthesize.md`: сгруппируй подходы, проверь
   каждый по реальному коду, отсей нежизнеспособные и уже испробованные, покажи
   разногласия, дай рекомендацию с вариантами. Консенсус — сигнал, не вердикт.
   Синтез идёт **после** файла находок: пользователь принимает решение, уже имея
   полный материал. В чат — только итоговая рекомендация по формату; сырые логи
   не пересказывай.

7. **(Опционально) Реализация.** Этот скилл только предлагает. Выбор варианта
   пользователь делает **обычным сообщением в чате** — не через интерактивное
   меню. После выбора передай вариант в `/glm implement` или `/codex implement`,
   либо реализуй сам.

8. **Очистка.** `rm -rf /tmp/multi-opinion/` — сохранённый файл находок
   (`council-*.md` в рабочей папке) не трогай.

## Безопасность и конфиденциальность

- Советники получают **описание проблемы и доступ на чтение к коду** в корне
  контекста и **отправляют это наружу**: codex → OpenAI, claude → Anthropic,
  glm → Z.ai (**китайский провайдер**), gemini → Google (через Antigravity
  `agy`; теперь тоже инспектирует код в корне контекста). Если у GLM
  включены research-инструменты (скилл `glm`), его веб-поиск / чтение страниц /
  `zread`-запросы тоже уходят в Z.ai, а через фолбек-инструменты поиска/чтения —
  в Tavily и Firecrawl (США).
- Read-only обеспечивается **по-разному**: у codex (sandbox) и glm (агент
  `plan`) — на уровне инструмента; у claude и gemini — только инструкцией в
  промпте («ничего не правь»), без жёсткого флага (у gemini `--sandbox`
  ограничивает терминал, но не правку файлов). Не считай их песочницей.
- Несмотря на read-only, **не созывай совет на NDA / ПДн / секретных
  репозиториях** — содержимое всё равно уходит сторонним провайдерам. Для
  чувствительного — ограничься локальным Claude или вообще не выноси наружу.
- **Редакция секретов в билдере — косметическая**: она чистит лишь частые
  шаблоны токенов в `prompt.md`/`problem.md` (и пропускает многие форматы) и
  **не трогает файлы под `context_root`**. Ответственность за то, чтобы реальные
  ключи не попали в текст проблемы и в корень контекста, — на тебе: держи
  секреты в `.env` (git-ignored) и не указывай `context_root` на дерево с ними.
- **Одна сессия `multi-opinion` на хост за раз.** Рабочая папка
  `/tmp/multi-opinion` общая — параллельный запуск затрёт чужие результаты.

## Быстрая навигация

| Шаг | Файл |
|-----|------|
| Сборка пакета, проверка CLI, команды советников | `references/cli-commands.md` |
| Правила синтеза предложений и формат отчёта | `references/synthesize.md` |

## Жёсткие правила

- **Сначала материал, потом решение.** Файл с полными находками (шаг 5)
  сохраняется в рабочей папке задачи и путь показывается ДО синтеза; не проси
  пользователя выбирать вариант, пока он не получил файл и отчёт.
- **Никаких интерактивных меню (AskUserQuestion) в этом скилле.** Выбор подхода —
  обычным сообщением в чате.
- **В чат — только структурированный отчёт (рекомендация + варианты).**
  Сырой stdout советников (`.out` / `.status` / `.meta`), `prompt.md`,
  `problem.md` и прочее содержимое `/tmp/multi-opinion/` — внутреннее сырьё.
  Никогда не вываливай его в чат и не пересказывай дословно; полные находки
  живут в файле шага 5, в ответ идёт только синтез по формату из
  `references/synthesize.md`.
- **Это предложения, а не вердикты.** Проверяй каждое по реальному коду перед
  тем, как рекомендовать или реализовывать.
- **Один советник = слабый сигнал; консенсус = сильнее, но не доказательство.**
- **Не реализуй молча.** Сначала рекомендация и выбор, потом реализация.
- **Уважай «что уже пробовали».** Не предлагай по кругу отвергнутое без новой
  причины.
- **Один советник — один вызов Bash.** Не объединяй CLI в одну строку.
