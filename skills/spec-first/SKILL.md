# spec-first — арка «Спека-первая»: идея → спека → план → реализация

Один пайплайн для крупных задач из трёх стадий. Каждая стадия полностью описана в своём файле в `stages/` — этот файл только маршрутизирует.

## Как работать

1. Определи стадию — по запросу пользователя и состоянию проекта (таблица ниже).
2. Прочитай файл этой стадии из `stages/` **целиком** и выполняй только его.
3. Стадии связаны через файлы-артефакты: спека `specs/<что-делает>.md` → план `plans/<задача>.md`. Никакого другого состояния между стадиями нет — каждую стадию можно (и лучше) начинать в свежей сессии.

## Маршрутизация

| Запрос пользователя | Состояние проекта | Стадия | Файл |
|---|---|---|---|
| «собери спеку», «составь ТЗ», «обмозгуем», новая крупная задача | спеки ещё нет | 1 — spec-interview | `stages/1-spec-interview.md` |
| «составь план», «разбей задачу на этапы» | есть одобренная спека в `specs/` | 2 — make-plan | `stages/2-make-plan.md` |
| «делай по плану», «реализуй с проверкой» | есть одобренный план в `plans/` | 3 — build-with-review | `stages/3-build-with-review.md` |

Если запрос противоречит состоянию (просят план, а спеки нет; просят реализацию, а плана нет) — **не пропускай стадии**: скажи, какой стадии не хватает, и предложи начать с неё.

## Жёсткие правила (для всех стадий)

- Одна сессия — одна стадия. Переход между стадиями — повод для новой сессии: артефакты лежат файлами, ничего не теряется.
- Гейты одобрения пользователем в конце стадий 1 и 2 обязательны и не пропускаются.
- `scripts/` и `prompts/` в корне этого скилла принадлежат стадии 3 — их запуск и шаблоны описаны в её файле.
