# Промпт финального ревью (whole-branch)

> **Модель: самая сильная доступная (дефолт — Opus). Это финальное ревью всей работы по плану.**

---

## Назначение

Финальный контроль качества всей реализации — **после того как все этапы плана завершены**.
Ты смотришь на работу целиком, а не на один этап.

---

## Входные данные

1. **Review-package диапазона `СТАРТ..HEAD`** — полный дифф от момента начала работы до текущего состояния.
   - `СТАРТ` берётся **из блока `## Прогон` плана** строкой `СТАРТ: <sha>`, записанной контроллером в начале прогона.
   - **Не реконструировать `СТАРТ` по памяти** — иначе можно молча проревьюировать только часть работы.
   - Если git недоступен — все файлы, произведённые в ходе работы по плану.

2. **Свод Minor** — список Minor-находок, накопленных за все этапы в блоке `## Прогон` плана. Задача финального ревью —
   **отсортировать свод Minor на триаж**: что стоит исправить до сдачи, что можно принять как есть, что закрыть как устаревшее.

---

## Код vs текст

Если реализация содержит код — проверяй его **код-рубрикой** субагента-критика (`critic-prompt.md`):
корректность, качество, тесты (проверяют реальное поведение, а не моки), архитектура, безопасность, готовность.

Для текстовых/документных артефактов — текст-рубрика и те же нормы честности и вердикты, что у `critic-prompt`:
`✅` / `❌` / `⚠️` + Critical / Important / Minor.

---

## Что проверить

1. **Целостность:** все этапы плана выполнены? Нет пробелов между этапами?
2. **Критерии приёмки из спеки:** каждый проверяемый пункт SPEC.md закрыт в итоговом результате?
3. **Глобальные ограничения:** инварианты плана соблюдены через всю работу?
4. **Блок «НЕ делаем»:** ничего лишнего не появилось в ходе реализации?
5. **Свод Minor:** какие из накопленных Minor стоит исправить сейчас?
6. **Регрессии между этапами:** поздние этапы не сломали то, что работало после ранних?

---

## Если финальное ревью нашло проблемы

Запустить **один** фиксер на весь список находок — не по отдельному фиксеру на каждую находку.

**Почему один:** каждый фиксер заново тянет контекст, прогоняет тесты, делает коммит.
Множественные фиксеры — это множественные перезагрузки контекста и риск конфликтов между правками.

Порядок работы фиксера:
1. Получить полный список Critical / Important / Minor-к-исправлению за один раз.
2. Исправить всё в одном проходе.
3. Прогнать проверки.
4. Один коммит на все правки.
5. Записать результат в report.

---

## Доказательства готовности

**«Готово»** — только с доказательствами:
- Для кода: прогоны тестов зелёные, smoke-проверка пройдена.
- Для документа: каждый критерий приёмки проверен и отмечен.
- Для смешанного: оба типа доказательств.

Не заявлять «всё готово» без исполняемых или верифицируемых подтверждений.

---

## Формат вывода

**Вердикт соответствия** (`✅` / `❌` / `⚠️`) + краткое обоснование.

**Результаты триажа свод Minor:**
- Исправить до сдачи: …
- Принять как есть (почему): …
- Закрыть как устаревшее (почему): …

**Находки финального ревью** (Critical / Important / Minor с file:line, если есть):
- …

**Вердикт:** Одобрено | Нужны правки | Нужен фиксер (с описанием задачи фиксера).

---

После принятия финального ревью — итоговый отчёт пользователю:
что сделано, что осталось в бэклоге, какие Minor остались открытыми.
