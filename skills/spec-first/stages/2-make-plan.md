
# Стадия 2: make-plan — спека → план реализации

Это **вторая стадия арки «Спека-первая»**: спецификация → **план** → реализация с проверкой.

Твоя задача здесь — превратить одобренную спеку в чёткий план-файл с декомпозицией на проверяемые этапы. Скилл **сам пишет план-документ** — по принципу мелкой проверяемой декомпозиции — и записывает его напрямую в `plans/<задача>.md`.

**Хендофф в конце этой стадии:** когда план одобрен, следующий шаг — стадия 3, `build-with-review` («делай по плану»).

---

## Почему НЕ используем встроенный Plan Mode

Встроенный Plan Mode **не подходит для этого пайплайна**:

- Он перетирает инструкции скилла **собственным системным промптом** (который явно «перекрывает любые другие инструкции»), из-за чего схема плана, контракт этапа и обязательный self-review теряются.
- Он ведёт свой многофазный воркфлоу и **пишет рандом-именованный файл** (например, `plans/vivid-drifting-kahan.md`) — вместо смыслового kebab-имени из нашей схемы.
- Он капризничает на не-кодовых и документных задачах (его `ExitPlanMode` описан «не для исследовательских/документных задач»).

Поэтому `make-plan` **сам пишет план-файл** — полностью контролирует схему, имя файла и контент. Plan Mode — отдельный инструмент Варианта 1 (небольшие задачи / новички), не часть этого пайплайна.

---

## Установка «исполнитель без контекста»

**Ключевой принцип при составлении плана:**

> План пишется в расчёте на исполнителя, который **не знает ни проект, ни предметную область** и читает только то, что ему дали.

Отсюда — точные пути к файлам, конкретные команды и ожидаемый вывод, без «разберись по ходу», «добавь аналогично предыдущему» или ссылок на «что мы обсуждали».

---

## Чеклист из 7 шагов (агент заводит задачу в todo на каждый пункт)

### Шаг 1 — Найти и прочитать спеку

Спеку указывает пользователь: «составь план из `specs/<что-делает>.md`» → берёшь этот путь. Если путь не назван — посмотри папку `specs/`:
- ровно одна спека → возьми её (кратко подтверди: «делаю план по `specs/…`?»);
- несколько → покажи список и спроси, по какой делать план;
- ни одной → предложи сперва собрать спеку (стадия 1 — `spec-interview`).

**Молча не угадывай** — при неоднозначности всегда уточняй у пользователя.

Прочитать спеку **полностью**, включая поля калибровки:
- `Технический профиль пользователя` (нетех / смешанный / тех) — определяет, насколько технично формулировать этапы и нужны ли git-операции.
- `Ожидаемый тип результата` (документ / код / смешанный) — определяет гранулярность этапов и маршрут критика.
- Блок «НЕ делаем» — войдёт в «Глобальные ограничения» плана дословно.
- Критерии приёмки — каждый критерий должен лечь на конкретный этап.

### Шаг 2 — Карта файлов/артефактов (ДО этапов)

Прежде чем делить на этапы — зафиксировать, **что создаём или меняем**:

- **Для код / смешанный:** список ключевых файлов с зоной ответственности каждого (`путь/файл — за что отвечает`).
- **Для документ:** какие выходные файлы или разделы производит эта работа.

Это «File Structure» решений: здесь принимается декомпозиция, а не внутри описания этапов. Карта войдёт в блок `## Карта файлов/артефактов` план-файла.

### Шаг 3 — Декомпозиция на этапы

Разбить работу на этапы, соблюдая **контракт этапа** (см. раздел ниже).

Ориентир по **гранулярности** зависит от калибровки:

**Для `код` / `смешанный`** — дробить мельче и конкретнее (мелкие проверяемые шаги):
- Точные файлы, которые трогаем/создаём.
- Конкретные команды и ожидаемый вывод.
- Для «код с тестами» — отдельными шагами **RED** (написать падающий тест → прогнать → убедиться, что падает) и **GREEN** (минимальная реализация → прогнать → проходит) + коммит.

**Для `документ`** — достаточно уровня этапа:
- Что производим на выходе + «Проверка» = сверка с критериями приёмки.
- Каждый документный этап обязательно называет **источники-входы** и ожидание «трасса источник→результат»: критик документа сверяет результат с источниками, а не только с критериями.

**Финальный этап — `CLAUDE.md` под фактическое состояние** (для `код` / `смешанный`). Последним этапом плана всегда ставь создание или актуализацию проектного `CLAUDE.md` — **в конце, по факту реализованного** (не план):
- `CLAUDE.md` нет (новый проект или существующий без него) → **создать**: назначение, структура, стек, конвенции, как запускать/тестировать.
- `CLAUDE.md` уже есть (дорабатываем проект) → **актуализировать**: внести изменения, убрать устаревшее.

Это **описание текущего состояния, а не журнал изменений**. Для `result_type = документ` этап опускается.

### Шаг 4 — Записать план-файл напрямую

Записать план в `plans/<задача>.md` — **осмысленное имя в kebab-case** (например, `plans/organise-notes-into-handbook.md`).

Использовать схему ниже (раздел «Схема план-файла»). Без рандом-имён, без правки `.claude/settings.json` — скилл **сам контролирует путь и имя**.

Ключевое: план **самодостаточен** и несёт состояние прогона в себе. В шапке проставь поле `**Спека:**` с путём к спеке (по нему `build-with-review` найдёт требования). Добавь пустой каркас блока `## Прогон` (СТАРТ ещё не известен — ставь `—`; свод Minor пуст) — его заполнит `build-with-review`. Никаких внешних файлов-состояния скилл не создаёт.

### Шаг 5 — Self-review плана (агент сам, чинит inline)

После записи — проверить план свежим взглядом, **до показа пользователю**:

1. **Покрытие:** каждый критерий приёмки из `SPEC.md` ложится на конкретный этап? Перечислить пробелы — и закрыть.
2. **Скан плейсхолдеров:** нет «TBD / добавь обработку ошибок / сделай аналогично этапу N / разберись по ходу»? Заменить конкретикой.
3. **Согласованность:** имена файлов, типы, сигнатуры одинаковы между этапами? Ссылки на файлы совпадают с картой файлов?

Найденное — чинить прямо в файле. Self-review — **обязательный шаг, не пропускается**.

### Шаг 6 — Ревью плана чистым контекстом (обязательный гейт)

Натравить **критика с чистым контекстом** на сам план-файл (и `SPEC.md` для сверки).

**Почему это гейт, а не опция:** слабый план «правит» весь автономный прогон `build-with-review`. Лучше потратить 2 минуты здесь, чем переделывать реализацию из-за размытых этапов или пропущенных требований.

Дать критику следующий промпт (вставить дословно):

```
Ты — ревьюер плана с чистым контекстом. Проверь, что план полон и готов к реализации.
Вход: план-файл + SPEC.md (для сверки).
Проверь по категориям:
- Полнота: TODO, плейсхолдеры, незавершённые этапы, пропущенные шаги.
- Соответствие спеке: план покрывает требования SPEC, нет разрастания объёма.
- Декомпозиция: у этапов чёткие границы, шаги выполнимы.
- Реализуемость: сможет ли исполнитель пройти план, не застряв?
Калибровка: флагай только то, что вызовет реальные проблемы при реализации
(пропущенные требования, противоречивые шаги, плейсхолдеры, слишком размытые этапы).
Мелкие формулировки и «было бы неплохо» — не повод блокировать.
Верни: Статус (Одобрено | Есть проблемы) + список проблем [этап X, шаг Y]: суть — почему важно + рекомендации (не блокирующие).
```

**Разбор находок — не слепое принятие.** Критик — советник. По каждой находке реши **с технической строгостью, без формального поддакивания**, реальна ли она:
- обоснованная и существенная → внести правку в план-файл;
- ложное срабатывание / не согласен → **не править**, зафиксировать одной строкой почему.

Что отклонил или в чём сомневаешься — не прятать, вынести на гейт (шаг 7), решает пользователь.

**Защита от петли.** Цикл «критик → правки» — **не более 2 раундов**; после правок можно один раз перепрогнать критика. Остались замечания и после второго раунда — **стоп**, на гейт (шаг 7) со списком открытого, без бесконечного ревью.

### Шаг 7 — Gate одобрения + хендофф

Показать план пользователю (или кратко: «план готов, критик проверил — вот файл»). Дождаться явного «ок».

Затем сказать:

> «План готов и проверен. Реализацию начинай в НОВОЙ сессии: скажи "делай по плану `plans/<задача>.md`" — и запустится стадия 3 (`build-with-review`).»

**Важно:** реализацию начинать в новой/свежей сессии — контекст планирования не тащим в цикл реализации.

## Схема план-файла (контракт — соблюдать дословно)

```markdown
# План: <название задачи>

> Для исполнителя: делай этап за этапом. После каждого — сохранение (коммит) и проверка критиком.

**Цель:** <одна фраза>
**Подход:** <2–3 фразы про то, как устроено>
**Профиль / тип результата:** <нетех|смешанный|тех> / <документ|код|смешанный>
**Спека:** specs/<что-делает>.md   <!-- источник требований; build-with-review читает спеку по этому пути -->

## Глобальные ограничения (инварианты — критик получает их ДОСЛОВНО)
<точные значения, форматы, связи компонентов («как X», «совпадает с Y») + перенос блока «НЕ делаем» из SPEC>

## Карта файлов/артефактов
- `путь/файл` — <за что отвечает>
- …

## Прогон
<!-- единственное место с лентой прогона; заполняет build-with-review. make-plan создаёт пустой каркас -->
- **СТАРТ:** —
- **Свод Minor:** (накапливается по ходу — для финального ревью)

## Этапы

### Этап 1: <название> — статус: todo
- **Делаем:** <что именно, конкретно>
- **Файлы/артефакты этапа:** <что трогаем/производим>
- **Критерии приёмки этапа:** <проверяемо>
- **Проверка:** <как убедиться, что готово — команда/действие/сверка>
- **Готово, когда:** <условие завершения>

### Этап 2: …  — статус: todo
…

### Этап N (финальный; для код/смешанный) — Создать/актуализировать `CLAUDE.md` — статус: todo
- **Делаем:** под фактическое состояние после работы — создать проектный `CLAUDE.md` (если его нет) либо актуализировать существующий: назначение, структура, стек, конвенции, запуск/тесты. Описание состояния, не журнал изменений.
- **Файлы/артефакты этапа:** `CLAUDE.md`
- **Проверка:** `CLAUDE.md` отражает реально реализованное; устаревшее убрано.
- **Готово, когда:** файл создан/обновлён и закоммичен.
<!-- для result_type = документ этот этап опускается -->

## Бэклог (вне основной линии)
- <идея/доработка, не входящая в этапы>
```

---

## Контракт этапа

Каждый этап обязан соответствовать всем пяти пунктам:

- **(а) Помещается в одну сессию** — объём реальный, не «переделать всё».
- **(б) Завершается до осязаемой точки** — что-то конкретное создано, изменено или проверено.
- **(в) Проверяем** — у этапа есть свои критерии приёмки и секция «Проверка» с конкретным действием.
- **(г) Имеет статус** — одно из: `todo | in progress | done`. Все этапы начинают с `todo`.
- **(д) Без плейсхолдеров** — конкретные файлы, действия, проверки. Ни одного «TBD», «добавь потом», «аналогично этапу N».

Блок «Глобальные ограничения», карта файлов, поле `**Спека:**`, каркас `## Прогон` и self-review плана — **обязательны** в каждом план-файле. Бэклог — **отдельным списком**, не смешан с этапами.

---

## Состояние прогона — в самом плане (без внешних файлов)

**Прогон** — один проход реализации плана стадией `build-with-review`: от стартового коммита до финала. Блок `## Прогон` в плане — его бортовой журнал (что сделано на каждом этапе, вердикты критика, нерешённое); `make-plan` кладёт пустой каркас, наполняет его `build-with-review` по ходу.

Состояние прогона целиком лежит в самом плане — отдельных файлов-состояния скилл не ведёт. Всё, что нужно следующей стадии:

- **`Спека:`** в шапке — путь к спеке (`build-with-review` читает требования по нему).
- **`## Прогон`** — каркас под ленту прогона (`СТАРТ`, по-этапная лента, свод Minor); его заполняет `build-with-review`.
- **Статус этапа** инлайн на заголовке (`— статус: todo`) — все этапы стартуют с `todo`.

Так несколько прогонов (по разным планам) не конкурируют за общий файл, а в новой сессии пользователь просто указывает нужный план.

---

## Контрольный список перед хендоффом

Перед тем как показать план пользователю, убедиться:

- [ ] Прочитал спеку полностью, включая калибровку и «НЕ делаем»
- [ ] Карта файлов/артефактов составлена (ДО этапов)
- [ ] Каждый критерий приёмки SPEC покрыт конкретным этапом
- [ ] Нет плейсхолдеров («TBD», «разберись по ходу», «аналогично этапу N»)
- [ ] Гранулярность соответствует калибровке (код — конкретнее; документ — уровень этапа с источниками)
- [ ] Для код/смешанный: последним этапом — создание/актуализация проектного `CLAUDE.md` под фактическое состояние
- [ ] Блок «Глобальные ограничения» включает «НЕ делаем» из SPEC дословно
- [ ] Поле `**Спека:**` проставлено; каркас `## Прогон` добавлен (СТАРТ `—`, свод Minor пуст)
- [ ] Self-review (шаг 5) пройден, найденное исправлено
- [ ] Ревью критиком (шаг 6) пройдено, замечания исправлены
