# build-with-review

Третий и финальный скилл арки **Спека-первая**: автономно реализует готовый план по этапам, выполняя после каждого этапа проверку независимым критиком на чистом контексте. Работает на месте (in-place), без веток и worktree, но с полным сохранением механик автономности, контекст-гигиены и честной самопроверки.

---

## 1. Назначение и старт в новой сессии

**Цель:** автономно, без вопросов между этапами, реализовать план, выдавая после каждого этапа пакет для независимого критика. Всё происходит in-place — никаких веток, никакого worktree.

**Старт:** всегда в **новой, свежей сессии** — контекст планирования не тащим. Контроллер-агент (ты) работает по принципу «нового сотрудника»: входит и читает документы с нуля.

**Какой план реализуем — указывает пользователь.** «делай по плану `plans/<задача>.md`» → берёшь этот путь. Если путь не назван — посмотри папку `plans/`:

- ровно один план → возьми его;
- несколько → покажи список и спроси, какой реализовать;
- ни одного → спроси у пользователя (плана ещё нет — сперва `make-plan`).

**Молча не угадывай.** Спеку находишь по полю `**Спека:**` в шапке плана — отдельный указатель не нужен.

Порядок чтения на старте:

1. `CLAUDE.md` проекта (глобальный + проектный) — среда, конвенции, стек.
2. Файл плана (`plans/<задача>.md`) — этапы, карта файлов, Глобальные ограничения, блок `## Прогон`, поле `Спека:`.
3. Файл спеки (путь из поля `Спека:` плана) — цель, ограничения, критерии приёмки.

**Состояние прогона — в самом плане, без внешних файлов** (ни singleton `progress.md`):

- **Калибровка** (`профиль`, `тип результата`) — в шапке плана (и в спеке).
- **Статус каждого этапа** — инлайн на заголовке (`— статус: todo|in progress|done`).
- **Лента прогона** (`СТАРТ`, по-этапные строки, свод Minor) — в блоке `## Прогон` плана.

Текущий этап не хранится отдельным числом — это **первый этап со статусом ≠ `done`**. Прогресс пишешь прямо в план по ходу выполнения.

**Параллельные прогоны.** Per-plan состояние снимает гонку за общий файл: `spec-interview` и `make-plan` можно спокойно гонять рядом с этим прогоном (они лишь пишут разные файлы в `specs/`/`plans/`). Чего **не** делай — два `build-with-review` одновременно в одном рабочем каталоге: они делят git-индекс и file-map-гард начнёт мешать. Нужны две реализации сразу — каждой свой каталог/копию репозитория.

## 2. Pre-flight (перед циклом)

Три обязательных проверки до старта цикла.

### 2.1. Скан плана на конфликты

Один раз просматриваешь план целиком: есть ли этапы, противоречащие друг другу, или требования, которые критик-рубрика заведомо посчитает дефектом? Найденное собираешь **одним батч-вопросом** пользователю до старта. Если противоречий нет — продолжаешь молча.

### 2.2. Git-гигиена

Проверяешь `git status`. Правило: чужие незакоммиченные правки **не подмешиваешь** (коммитишь только файлы своего этапа точечным pathspec из карты этапа — никогда не `git add -A` слепо). При «грязном» дереве спрашиваешь простым языком: «у тебя есть несохранённые правки из другой работы — сохранить их отдельной точкой?». Никогда не коммитишь секреты (`.env`, ключи, токены) и тяжёлые артефакты (сборки, зависимости).

### 2.3. Ledger — блок `## Прогон` в плане

Лента прогона живёт в блоке `## Прогон` самого плана (виден студенту, коммитится вместе с этапами, переживает компакт). На старте:

- Если `СТАРТ` ещё `—` (новый прогон) — впиши `СТАРТ: <sha>` (коммит до первого этапа). Эта строка нужна финальному whole-branch ревью.
- Если прогон продолжается (после компакта / в новой сессии) — читаешь `## Прогон` и статусы этапов. Этапы со статусом `done` в плане **не перезапускаешь** — продолжаешь с первого не-`done`. Это защита от самого дорогого провала: переисполнение готовой работы.

## 3. Автономный цикл (10 шагов)

**Это один непрерывный цикл, между шагами остановок нет.** Шаги 1–9 контроллер прогоняет на каждый этап, шаг 10 возвращает к шагу 1 для следующего — всё автономно, без вопросов между этапами. Единственные остановки: статус BLOCKED, который нельзя разрешить самостоятельно; реальная двусмысленность, мешающая продолжить; план исчерпан.

### Шаг 1 — Взять следующий этап

Найти первый этап со статусом **≠ `done`** (не «первый `todo`»: после прерывания этап мог остаться `in progress`). Дальше по его статусу:

- `todo` → пометь `in progress` (правишь инлайн-статус на заголовке этапа) и делай с нуля;
- `in progress` (прерванный прогон) → **возобнови** его по строке этапа в `## Прогон` + `git log`, а не пропускай и не начинай заново вслепую.

Отдельного `current_stage` нет — текущий этап определяется по статусам в плане.

### Шаг 2 — task-brief

Запустить скрипт вырезки этапа в отдельный файл:

```bash
bash <путь-к-скиллу>/scripts/task-brief <plan> N
```

Скрипт ищет заголовок `### Этап N` и кладёт текст в `.spec-first/sdd/<слаг-плана>/etap-N-brief.md` (слаг выводится из имени плана — у каждого прогона своя папка, два прогона не пересекаются). Исполнитель читает только свой файл — текст этапа не идёт через контекст контроллера (контекст-гигиена). Блок `## Прогон` в бриф не попадает (он стоит до `## Этапы`), поэтому требования остаются чистыми.

### Шаг 3 — Исполнитель (субагент)

Запустить **свежий субагент** по шаблону `prompts/implementer-prompt.md`. Диспатч содержит:

- Одна строка «где этот этап в проекте» (краткий ориентир).
- Путь к brief-файлу: «читай первым — это твои требования, значения брать дословно».
- Интерфейсы и ключевые решения прошлых этапов, которых brief знать не может.
- Путь к report-файлу и контракт отчёта (что писать, 4 статуса).

**В диспатч не вставлять накопленную историю прошлых этапов** — только это задание. Исполнитель работает с чистым контекстом.

Исполнитель обязан:

- Сделать этап согласно brief.
- Выполнить self-review свежим взглядом (полнота / качество / дисциплина-YAGNI / тесты) и починить до сдачи.
- Написать **полный отчёт** в report-файл.
- Вернуть контроллеру **≤15 строк**: статус + коммиты + одна строка про тесты + опасения + путь к report.

Для **простых задач** (калибровка схлопывает цикл): контроллер сам делает этап, затем запускает субагент-критик.

### Шаг 4 — Обработка 4 статусов исполнителя


| Статус                 | Действие                                                                                                                                                                                                                                           |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **DONE**               | Переходи к шагу 5 (сохранение) → шагу 6 (review-package) → шагу 7 (критик).                                                                                                                                                                        |
| **DONE_WITH_CONCERNS** | Прочитай опасения. Вопросы корректности/объёма — закрой до ревью (вернуть исполнителю или разрешить сам). Наблюдения-«симптомы» («файл растёт») — отметь в блоке `## Прогон` и продолжай.                                                          |
| **NEEDS_CONTEXT**      | Дай недостающее (файл, решение прошлого этапа, пояснение) и перезапусти исполнителя.                                                                                                                                                               |
| **BLOCKED**            | Оцени: проблема контекста → дать и перезапустить; нужна больше «соображалки» → перезапустить более сильной моделью; задача велика → разбить этап; план неверен → эскалация пользователю. **Никогда** не гоняй ту же модель повторно без изменений. |


### Шаг 5 — Сохранение + радиус поражения

**Запомни BASE-коммит** до начала этапа (не пересчитывай потом — `BASE` нужен review-package и файл-карта-гарду).

После работы исполнителя:

1. **Git-коммит точечным pathspec** — файлы из «Файлы/артефакты этапа» **плюс сам файл плана** (в нём обновлены статус этапа и строка `## Прогон` — это бухгалтерия прогона, коммитим вместе с этапом, чтобы `git log` и план были согласованы):
  ```bash
   git add путь/к/файлу1 путь/к/файлу2 plans/<задача>.md
   git commit -m "Этап N: <что сделано>"
  ```
2. **Файл-карта-гард:**
  ```bash
   git diff --name-only BASE HEAD
  ```

   Любой файл вне карты этапа = дефект → эскалация (так радиус поражения держится на уровне файлов без веток). **Исключения** — сам файл плана и файл спеки: план обновляется по ходу (бухгалтерия прогона), спека read-only; их в гарде игнорируешь.
3. **Снапшот всегда** — копия изменённых файлов этапа в `.spec-first/snapshots/<слаг-плана>/<этап>/` (без `node_modules`, `.next`, зависимостей). Снапшот делается при любом исходе — он единственный откат при отсутствии git.
4. **Откат этапа** = `git revert` его коммитов (не `reset --hard` — он стёр бы незакоммиченное пользователя в других окнах). Нет git → агент предлагает `git init`; снапшот — единственный откат. Без git правки плана сохраняются на диске.

### Шаг 6 — review-package

Собрать пакет для критика:

```bash
bash <путь-к-скиллу>/scripts/review-package BASE HEAD <слаг-плана>
```

Скрипт собирает `git log` + `git diff --stat` + `git diff -U10` за `BASE..HEAD` в один файл под папкой прогона `.spec-first/sdd/<слаг-плана>/`. **BASE — записанный до этапа, не `HEAD~1`** (иначе теряются все коммиты этапа кроме последнего). Нет git → скрипт пропускается, критику отдаются сами файлы этапа.

### Шаг 7 — Критик с чистым контекстом

Запустить критика по маршруту (см. раздел 10). Критику дать **три файла**:

1. `etap-N-brief.md` (задание этапа).
2. `etap-N-report.md` (отчёт исполнителя).
3. `review-package` (дифф; нет git → произведённые файлы этапа).

**Плюс блок «Глобальные ограничения» из плана — дословно** (это «линза внимания» критика, от неё нельзя отступать).

Критик возвращает:

- Вердикт: ✅ соответствует / ❌ проблемы / ⚠️ не проверяется по этому диффу.
- Сильные стороны.
- Находки по важности: **Critical / Important / Minor** с ссылками `file:line`.
- Вердикт качества (одна строка).

### Шаг 8 — Разбор вердиктов

**⚠️ «не проверяется по этому диффу»** — контроллер разрешает сам (у него кросс-этапный контекст, которого критику не дали). Подтвердился реальный разрыв → трактуй как проваленную спеку, верни исполнителю + повторное ревью.

**Critical / Important** — запустить **фиксер**: правит находки с технической строгостью, без формального поддакивания; перепрогоняет покрывающие тесты; дописывает результат в report → новое сохранение → повторное ревью. **Лимит: 2–3 попытки на этап** → не закрыто → стоп + эскалация со списком нерешённого. Видимый счётчик «попытка N из 3» пишется в строке этапа блока `## Прогон`.

**Minor** — записать в свод Minor блока `## Прогон`, **не чинить по ходу**; отдать его финальному ревью на триаж.

**plan-mandated** (находка конфликтует с тем, что предписывает план) — решает **пользователь** через эскалацию-рекомендацию: показать находку, текст плана, свою рекомендацию. Не отбрасывать находку только потому, что «план так велит».

### Шаг 9 — Ledger (блок `## Прогон`)

По каждому этапу дописать/обновить строку в блоке `## Прогон` плана:

```
- **Этап N:** коммиты [[ORCA_RAW_HTML_INLINE:%3Cbase7%3E]]..[[ORCA_RAW_HTML_INLINE:%3Chead7%3E]] | попытки: N/3 | вердикт: ✅/❌/⚠️ | нерешённое: ... | next: ...
```

Готовый этап (после того как критик принял результат):

1. Поставить статус этапа в плане: `in progress` → `**done**` (инлайн на заголовке этапа).
2. Обновить его строку в `## Прогон`: `вердикт: ✅ | нерешённое: — | next: —`.

Блок `## Прогон` — это recovery-карта после компакта. После компакта доверяй `## Прогон` + статусам этапов + `git log`, а не своей памяти.

### Шаг 10 — Следующий этап / финал

Перейти к шагу 1 для следующего этапа со статусом ≠ `done` (отдельный `current_stage` не ведём — он читается из статусов плана).

**По исчерпании плана — финальное whole-branch ревью:**

1. Собрать review-package диапазона `СТАРТ..HEAD` (`СТАРТ` из блока `## Прогон` — не реконструировать по памяти, иначе отревьюируешь лишь часть работы).
2. Запустить критика по шаблону `prompts/final-review-prompt.md` (для кода — с код-рубрикой) на всём диапазоне + **свод Minor** из `## Прогон` на триаж.
3. Если финал вернул находки — **один** фиксер на весь список (не по фиксеру на каждую находку: каждый заново тянет контекст и гоняет тесты).
4. Заявлять «готово» — только с доказательствами прогонов, не на словах.
5. Итоговый отчёт пользователю — как при сдаче готовой работы: что сделано, что в бэклоге, какие Minor остались открытыми.

---

## 4. Эскалация — через рекомендацию, не открытый вопрос

Любая остановка цикла (BLOCKED, plan-mandated, исчерпан лимит фиксов) подаётся так:

1. Контекст простыми словами — что произошло, почему не можем продолжить.
2. **Своя рекомендация с обоснованием** — что лучше сделать и почему.
3. Три варианта на выбор пользователю (формулировки — как сказать ему; *курсив в скобках — для тебя, пользователю не показываешь*):
  - **1)** отменить сделанное на этом этапе и попробовать иначе; *(откат = `git revert` коммитов этапа, как в Шаге 5 — не `reset --hard`; затем этап заново)*
  - **2)** счесть замечание мелким (не блокирующим) и продолжить; *(записать находку в свод Minor блока `## Прогон`, этап закрыть как `done`, идти дальше)*
  - **3)** объясни ошибку — подскажу текстом. *(дождаться подсказки пользователя и продолжить этап с её учётом)*

Видимый счётчик «попытка N из 3» (пишется в блоке `## Прогон`) — чтобы цикл не зациклился после компакта.

---

## 5. Нормы честности критика

Обязательны — вносятся в промпт критика дословно.

- **«Не доверяй отчёту»** — отчёт исполнителя это непроверенные утверждения; судишь по результату/диффу. «Оставил по YAGNI / специально упростил» — это самооценка, она **не понижает** серьёзность находки.
- **«Не суди за критика»** — контроллер не пишет критику «не флагай это», «максимум Minor», «план так решил». Думаешь, что ложное срабатывание — пусть критик поднимет, а ты рассудишь в цикле.
- Критик **read-only** — не трогает дерево/индекс/HEAD; дифф читает из файла за один проход; по кодовой базе не ползает (точечная проверка — только под названный риск); тесты, которые исполнитель уже прогнал на этом же коде, не перепрогоняет.

## 6. Хелпер-скрипты и шаблоны промптов

Скилл везёт `scripts/` и `prompts/`. **Запускает и применяет их агент** — студент их не видит и не настраивает. Смысл — контекст-гигиена: тяжёлый текст (бриф этапа, отчёт исполнителя, дифф) выгружается в файлы, минуя контекст контроллера.

### Скрипты (`scripts/`)

Все три кладут артефакты в `.spec-first/sdd/<слаг-прогона>/` рабочей папки проекта (слаг = имя файла плана без расширения; у каждого прогона своя папка). Запускать через `bash <путь>`, где путь вычисляется **от базовой директории скилла** при вызове (скилл может стоять в project- или user-scope).

- `**scripts/sdd-workspace [СЛАГ]**` — вспомогательный: единый источник пути рабочей папки. Со слагом печатает `.spec-first/sdd/<слаг>`, без — базовый `.spec-first/sdd` (обратная совместимость). Работает **без git** (без аварийного падения на `git rev-parse`): кладёт папку относительно корня проекта и добавляет `.gitignore` только при наличии git.
- `**scripts/task-brief <plan> N**` — вырезает текст одного этапа из плана. Ищет заголовки `### Этап N`; слаг выводит из имени плана и кладёт бриф в папку прогона. Без git работает через no-git-фолбэк `sdd-workspace`.
- `**scripts/review-package BASE HEAD [СЛАГ]**` — собирает `git log` + `git diff --stat` + `git diff -U10` за `BASE..HEAD` в один файл для критика, в папку прогона. При отсутствии git — пропускается.

### Шаблоны промптов (`prompts/`)

- `**prompts/implementer-prompt.md**` — диспатч исполнителя: задание + brief + блок self-review + контракт отчёта + 4 статуса.
- `**prompts/critic-prompt.md**` — диспатч доменно-нейтрального критика (текст/смешанный): «не доверяй отчёту», рубрика (критерии приёмки + «НЕ делаем»), вердикты ✅/❌/⚠️ + Critical/Important/Minor с `file:line`.
- `**prompts/final-review-prompt.md**` — финальное whole-branch ревью (для кода — код-рубрика критика).

### Мост языков

Системные сообщения, ошибки и API часто на английском. Правило для всех субагентов: докладывать статус по-русски, но текст ошибки приводить **дословно** (по-английски), не переводя.

### Когда начинать новый чат

Состояние прогона живёт в плане (`## Прогон` + статусы этапов), поэтому продолжить в свежей сессии можно с любого незавершённого этапа без потерь. Решение «пора в новый чат» — за пользователем (заполненность контекста видит он, не ты); если сессия затянулась, между этапами можно предложить ему продолжить в новом чате.

## 7. Глубина реализации по калибровке

**Полный автономный режим** (дефолт для этой пачки): на каждый этап — отдельный субагент-исполнитель + субагент-критик + фиксер; контроллер не спрашивает, доводит план до конца (полностью автономно, in-place).

**Для совсем простых задач** калибровка может схлопнуть цикл: контроллер сам делает этап, затем запускает субагент-критик — но без тяжёлой церемонии в любом случае.

## 8. Таблица выбора модели

Правило: **самая дешёвая модель, что тянет роль; указывать ЯВНО** (иначе субагент наследует модель сессии — часто самую дорогую).


| Роль                                                                               | Модель     |
| ---------------------------------------------------------------------------------- | ---------- |
| Контроллер / основной агент                                                        | **Opus**   |
| Исполнитель — дословная транскрипция кода из brief (1–2 файла)                     | **Haiku**  |
| Исполнитель — код с суждением, многофайловость, или любая проза / синтез документа | **Sonnet** |
| Критик (соответствие + качество), нейтральный критик, финальное ревью              | **Opus**   |


**Haiku** — только когда в brief лежит готовый код для переноса; работа «по описанию» (проза, синтез документа, демо) идёт минимум на Sonnet.

Если Desktop не даёт назначать модель отдельному субагенту — все роли идут на доступной модели (изоляция контекста остаётся, экономия на модели — нет).

## 9. Проверка кода по артефакту

Требование зависит от **того, что произвёл этап**, а не от профиля пользователя.

- **Код с тестами** → полный TDD: исполнитель докладывает в report **RED/GREEN-доказательства** — команда + релевантный падающий вывод ДО (почему падение ожидалось); команда + проходящий вывод ПОСЛЕ. Критик их не перепрогоняет.
- **Код без тест-харнесса** → минимум **smoke-проверка**: код запускается и существующий набор тестов остаётся зелёным; greenfield — один представительный тест или хотя бы синтаксис-чек (`node -c`, `python -m py_compile`). Закрывает дыру «критик читает код, но ничего не запускалось».
- **Документ** → исполняемой проверки нет; «проверка» = сверка с критериями приёмки и секцией «Проверка» из плана.

«Совсем без проверки» — только для чисто документного пути.

При баге или падении теста исполнитель и фиксер сначала воспроизводят и находят причину, потом чинят — не латают наугад.

---

## 10. Маршрут критика по типу артефакта

Критик выбирается по тому, **что реально произвёл этап**, а не по тому, технарь ли пользователь.


| Что произвёл этап | Критик              | Механизм                                          |
| ----------------- | ------------------- | ------------------------------------------------- |
| Код / скрипт      | Код-ревьюер         | субагент-критик, код-рубрика (prompts/critic-prompt.md)              |
| Документ / текст  | Доменно-нейтральный | субагент-критик, текст-рубрика (prompts/critic-prompt.md)            |
| Смешанный         | Оба                 | тот же субагент-критик: код-рубрика на код + текст-рубрика на текст  |


*Нетехнарь, чей агент написал скрипт, всё равно получает код-ревью этого скрипта.*

По умолчанию — субагент (Desktop его поддерживает, автономность не рвётся). Открыть свежий чат вручную — крайний фолбэк, только если субагент недоступен.

## 11. Ledger + систематическая отладка

**Durable progress:** память диалога не переживает компакт — весь прогресс ведётся в плане (статусы этапов + блок `## Прогон`), который коммитится вместе с этапами. После компакта доверять `## Прогон` + статусам этапов + `git log`, а не своей памяти. Готовые этапы (статус `done` в плане) не перезапускать.

Блок `## Прогон` хранит по каждому этапу: диапазон коммитов `<base7>..<head7>`, число попыток, последний вердикт критика, нерешённые Critical/Important/Minor, next-action; плюс `СТАРТ` и свод Minor. Это полная recovery-карта.

**Баг или падение теста** → исполнитель и фиксер сначала воспроизводят и находят причину, потом чинят — не латают наугад.

## 12. Состояние прогона — в плане (без внешних файлов)

Никакого `.spec-first/current.json` и никакого singleton `progress.md`. Всё состояние прогона держится в самих артефактах:


| Что                                      | Где                             | Кто пишет                                |
| ---------------------------------------- | ------------------------------- | ---------------------------------------- |
| Калибровка (`профиль`, `тип результата`) | инлайн в спеке + шапка плана    | `spec-interview` / `make-plan`           |
| Путь к спеке                             | поле `**Спека:**` в шапке плана | `make-plan`                              |
| Статус этапа (`todo/in progress/done`)   | инлайн на заголовке этапа       | `make-plan` (todo) → `build-with-review` |
| `СТАРТ`, по-этапная лента, свод Minor    | блок `## Прогон` плана          | `build-with-review`                      |


Текущий этап = первый со статусом ≠ `done` (отдельным числом не хранится). Так несколько прогонов по разным планам не конкурируют за общий файл, а хендофф между сессиями стабилен: в новой сессии пользователь указывает план, всё остальное читается из него. План коммитится вместе с каждым этапом, поэтому состояние переживает даже жёсткий сброс.
