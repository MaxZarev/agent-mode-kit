# multi-review

## Назначение

Запустить несколько внешних AI-CLI (codex, gemini, claude, glm) параллельно по
одному и тому же артефакту, а затем **проверить каждую их находку по реальному
коду или тексту** и отклонить ложные, преувеличенные и переусложнённые — а не
просто пересказать пользователю всё, что выдали модели. Внешние модели часто
ошибаются и завышают серьёзность; задача ведущего агента — отсеять шум, а не
слепо передать его дальше. Базовый поток лёгкий: внешние ревьюеры получают
локальные пути и компактный промпт, после чего сами инспектируют репозиторий
или файлы. Не передавай большие склеенные файлы, если только конкретный CLI не
умеет читать локальные файлы и не нужна ограниченная по объёму выдержка.

Ценность по сравнению с одним ревьюером — **разнообразие слепых зон**. Риск —
**галлюцинированные, раздутые или переусложнённые находки**. Этот скилл
кодифицирует обе стороны.

Скилл самодостаточный: используй его, когда нужно просто кросс-ревью одного
артефакта — PR, спецификации, статьи, системного промпта, юридического
документа, маркетингового текста или любого другого текста — без полноценного
пайплайна подготовки к деплою.

## Когда использовать

Используй, когда пользователь просит:

- Мульти-ИИ / кросс-ревью / внешнее ревью кода (текущая ветка,
  незакоммиченный diff или конкретный файл).
- Второе мнение по спецификации, плану реализации или дизайн-документу.
- Независимое ревью некодового текста: статья, пост в блог, системный промпт,
  маркетинговый текст, юридический документ, README, RFC, черновик письма и т.д.

НЕ используй, когда:

- Изменение тривиальное (опечатка, однострочная правка конфига, мелкая
  редактура текста) — накладные расходы не оправданы.
- Пользователь попросил конкретного одного ревьюера («просто спроси codex»,
  «только Gemini») — вызови этот один CLI напрямую.
- Не установлен ни один внешний CLI (скилл деградирует и сам по себе даёт мало).

## Режимы

Скилл работает в одном из двух режимов. Определи режим по намерению
пользователя до начала любых действий; спрашивай только при реальной
неоднозначности.

### Режим кода (code mode)

Срабатывает, когда артефакт — это набор git-изменений: diff ветки,
незакоммиченные изменения или рабочее дерево.

Ревьюеры получают: корень проекта, режим охвата (scope), базовый ref, список
изменённых файлов и применимые рекомендации репозитория
(`AGENTS.md` / `CLAUDE.md` / `GEMINI.md`). Они должны сами выполнять
read-only инспекцию git/файлов локально. Codex, Claude и Gemini не должны
получать полный diff, вставленный в промпт — они сами инспектируют изменённые
файлы из корня контекста (gemini — через `--add-dir`, read-only по промпту +
`--sandbox`).

### Режим текста (text mode)

Срабатывает, когда артефакт — отдельный текстовый файл или вставленный контент,
не подкреплённый git-diff: спецификация, план, статья, промпт, юридический
документ, маркетинговый текст, RFC и т.д.

Ревьюеры получают: путь(и) к файлам (абсолютные), тип контента (article /
spec / plan / prompt / legal / marketing / generic), аудиторию и цель, если
пользователь их указал, и инструкцию открыть файлы локально. Codex, Claude и
Gemini не должны получать склеенный полный артефакт — они открывают файлы
локально (gemini — через `--add-dir`, read-only по промпту + `--sandbox`).

## Рабочий процесс

1. **Определи режим и цель.** Посмотри на запрос пользователя:
   - Упоминается ветка, diff, PR, незакоммиченные изменения или «изменения,
     которые я сделал» → **режим кода**.
   - Назван файл, который не является кодом, или вставлен текст → **режим
     текста**.
   - Назван markdown-спек/план, лежащий внутри репозитория кода → **режим
     текста** (ревьюим сам документ, а не репозиторий вокруг него).
   - Если действительно неоднозначно (например, «проверь это») — спроси один
     раз: код или текст.

2. **Собери лёгкий пакет ревью** в `/tmp/multi-review/`.
   - Режим кода: запусти `scripts/build-code-review-pack.sh` (см.
     `references/cli-commands.md`).
   - Режим текста: запусти `scripts/build-text-review-pack.sh` (см.
     `references/cli-commands.md`).

   Оба скрипта пишут готовый к использованию `prompt.md` и редактируют
   распространённые шаблоны секретов в любом сгенерированном вспомогательном
   материале. Режим кода удаляет временный полный diff перед выходом. Режим
   текста не создаёт склеенный полный артефакт.

   Если билдер не запускается (нет `bash` / `python` / `perl` — частый случай
   на нативном Windows), не останавливайся: перейди на ручной запасной путь из
   `references/fallback.md`.

3. **Используй сгенерированный промпт.** Не собирай вручную второй промпт из
   больших файлов. Билдер пакета уже записал `/tmp/multi-review/prompt.md`.

4. **Проверь доступные CLI.** Запусти блок проверки доступности из
   `references/cli-commands.md`. Должен быть доступен хотя бы один из codex /
   gemini / claude / glm. Пропусти и отметь те, что отсутствуют или не
   авторизованы. Если доступно ноль — остановись, сообщи пользователю и
   предложи установить любой из CLI по `references/install-clis.md`.

5. **Запусти ревьюеров параллельно.** Один **отдельный** вызов инструмента
   Bash на каждый CLI, `run_in_background: true`, `timeout: 900000` (15
   минут). Используй `scripts/run-external-reviewer.sh`, чтобы таймауты и
   изоляция сбоев по каждому ревьюеру были единообразными. Передавай корень
   контекста флагом `--cwd "$(cat /tmp/multi-review/context_root.txt)"`. Не
   объединяй CLI через `&&` в одном вызове — отклонённый или упавший вызов
   отменит остальные.

6. **Разбери каждую находку** по `references/triage.md`. Сначала прочитай
   `<reviewer>.out` каждого ревьюера — это **сырой лог CLI** (прогресс,
   tool-calls, размышления модели вперемешку с находками), а не готовый отчёт.
   Извлеки из него только находки; сам лог в чат не выводи. Не принимай находки
   на веру: проверяй каждое утверждение по фактическому артефакту (коду или
   тексту) прежде чем принять его. Отклоняй ложные, нерелевантные или
   переусложнённые находки. Метки серьёзности от внешних моделей — это
   гипотезы, а не вердикты.

7. **Полные находки — в файл (ДО отчёта).** Выпиши из `<reviewer>.out` каждого
   ревьюера **все** извлечённые находки (включая те, что ты потом отклонишь при
   разборе — с твоей пометкой «подтвердилось / не подтвердилось и почему») в
   один читаемый md-файл. Сырой лог CLI в файл не копируй — только находки.
   Собери его в `/tmp/multi-review/findings.md`, затем **перенеси в рабочую
   папку текущей задачи** (туда, где живут её рабочие файлы: для подготовки
   урока — папка сессии; иначе — корень проекта) под именем
   `review-<YYYY-MM-DD>-<тема>.md` и покажи путь пользователю.

8. **Отчёт.** Используй формат из `references/triage.md` — сгруппируй находки
   в «к исправлению», «только для сведения», «отклонено» и перечисли
   пропущенных ревьюеров с причинами. **Только этот структурированный отчёт
   идёт в чат пользователю.** Не вставляй в ответ сырой вывод ревьюеров
   (`.out`), `prompt.md`, diff, содержимое `/tmp/multi-review/` или дословный
   пересказ stdout — пользователь видит только разобранные находки; полная
   версия у него уже есть в файле шага 7. Решения о применении находок не
   запрашивай раньше, чем выданы файл и отчёт, и **не через интерактивные меню
   (AskUserQuestion)** — выбор обычным сообщением в чате. Язык отчёта
   по умолчанию: русский (если пользователь работает на другом языке — зеркаль
   его язык).

9. **Очистка.** `rm -rf /tmp/multi-review/` — сохранённый файл находок
   (`review-*.md` в рабочей папке) не трогай.

## Где лежат скрипты

Скилл не использует переменные окружения. Вызывай скрипты по абсолютному пути к
директории, куда скилл установлен — например `~/.claude/skills/multi-review`,
`~/.agents/skills/multi-review`, `~/.gemini/skills/multi-review` или
`<repo>/multi-review`. Рабочая директория пакета ревью всегда
`/tmp/multi-review`.

## Платформы и запасной путь

Скрипты — это bash на Unix-инструментах (`bash`, `git`, `python3`, `perl`).
Перед запуском оцени окружение:

- **macOS / Linux:** запускай скрипты напрямую (или через `bash <путь>`).
- **Windows:** запускай через **WSL** (работает как Linux) или **Git Bash**.
  Обёртка `run-external-reviewer.sh` сама пробует `python3` и `python`.
- **Нативный Windows без bash, либо скрипт упал из-за отсутствия `bash` /
  `python` / `perl`:** не останавливайся. Скрипты — это удобство, а не
  зависимость. Перейди на **ручной запасной путь** из `references/fallback.md`
  и выполни ту же работу своими инструментами (собрать цель через git/чтение
  файлов, составить промпт по схеме, запустить каждый CLI напрямую, разобрать
  находки).

Сигналы перейти на fallback: `bash` недоступен; билдер завершился ошибкой;
`run-external-reviewer.sh` записал статус `no_python`.

## Жёсткие правила

- **Сначала материал, потом решение.** Файл с полными находками (шаг 7)
  сохраняется в рабочей папке задачи и путь показывается ДО отчёта и любых
  вопросов «что применяем». Никаких интерактивных меню (AskUserQuestion) —
  выбор обычным сообщением в чате.
- **Финальный ответ — только структурированный отчёт.** Сырой stdout ревьюеров
  (`.out` / `.status` / `.meta`), `prompt.md`, diff и прочее содержимое
  `/tmp/multi-review/` — внутреннее сырьё. Никогда не вываливай его в чат и не
  пересказывай дословно; полные находки живут в файле шага 7, в ответ идут
  только разобранные находки по формату из `references/triage.md`.
- **Метки серьёзности от внешних моделей — гипотезы, а не вердикты.** `CRITICAL`
  от одного CLI — это лишь приор одной модели. Проверяй по артефакту прежде
  чем действовать.
- **Не вставляй тяжёлые пакеты в промпты ревьюеров.** Все четыре (codex, claude,
  gemini, glm) получают только `prompt.md` и инспектируют локальные файлы сами —
  gemini теперь тоже через `--add-dir` (read-only по промпту + `--sandbox`).
- **Никогда не применяй и не принимай находку, не прочитав сам артефакт.** Для
  кода — открой файл. Для текста — прочитай соответствующий фрагмент.
- **Никогда не добавляй абстракции, защитную валидацию, shim-ы обратной
  совместимости или «энтерпрайз»-паттерны**, которые `AGENTS.md` / `CLAUDE.md`
  проекта велит избегать — даже под меткой `CRITICAL`.
- **Один ревьюер, отметивший что-то, — слабый сигнал.** Консенсус сильнее, но
  тоже не доказательство — модели разделяют слепые зоны.
- **Для текстовых артефактов уважай голос и ограничения автора.** Не переписывай
  молча тон, регистр или структуру, которые автор выбрал намеренно. Вместо
  этого вынеси компромисс пользователю.
- **Когда неоднозначно — спроси пользователя.** Не применяй молча изменения к
  рабочему коду или тексту, который пользователь уже одобрил.

## Частые ошибки

- **Относиться к CLI как к авторитетам.** Они разделяют слепые зоны и
  галлюцинируют симметрично. Голос 3/3 за неверное исправление всё равно неверен.
- **Пересылать полный diff в режиме кода.** Это тратит контекст и заставляет
  ревьюеров переобучаться на изменённых кусках. Используй сгенерированный
  репозиторий-нативный промпт, чтобы ревьюеры сами открывали затронутые файлы
  и связанных вызывающих.
- **Запускать CLI пачкой в одном Bash-блоке.** Один отказ отменяет все. Всегда
  один CLI на один вызов инструмента Bash.
- **Авто-применять каждый CRITICAL.** Так рабочий код или текст превращается в
  раздутую прозу. Сначала разбери.
- **Пропускать проход по корню проекта в режиме кода.** Если ревьюер не может
  открывать файлы, любая кросс-файловая находка — спекуляция. Используй `--cd`
  (codex) и `--add-dir` (claude и gemini/`agy`). Все четыре ревьюера
  инспектируют файлы из корня контекста сами.
- **Просить ревьюеров чинить вместо того, чтобы сообщать.** Этот скилл
  read-only — ревьюеры описывают находки, ты (или отдельный шаг исправления)
  решаешь и применяешь.
- **Вываливать сырой вывод ревьюеров в чат.** `.out`-файлы содержат служебный
  лог CLI и многословные размышления модели, а не готовый отчёт. Пользователю
  идёт только разобранный структурированный отчёт — не пересказ stdout, не
  склейка четырёх выводов, не `prompt.md` и не diff.
