# Глобальные инструкции

## Как со мной общаться

- Я немного разбираюсь в разработке.
- Не используй сложный технический язык, делай пояснения простыми словами
- Любое найденное в интернете решение проверяй на безопасность и предупреждай о рисках.

## Ответы на вопросы

- Внимательно читай запрос пользователя. Отвечай ровно на то, что спросили — не подменяй вопрос похожим
- Если в вопросе есть уточнение места/области («в user scope», «в этом файле», «на сервере», «в проекте X») — СТОП. НЕ отвечай из закешированного контекста, списков из системного промпта или памяти. Сначала проверь именно это место инструментом (Glob, Read, Bash), и только потом отвечай на основе того, что нашёл. Пример нарушения: пользователь спрашивает «есть ли музыкальные скиллы в user scope?», а ты перечисляешь скиллы из системного промпта вместо того, чтобы проверить ~/.claude/skills/
- Если не уверен, о чём именно спрашивают — попроси уточнить, а не угадывай

## Списки и перечисления

- Когда перечисляешь варианты, опции или пункты в чате — ВСЕГДА используй цифры (1, 2, 3), НИКОГДА буквы (A, B, C) или римские цифры. Касается всех форм: инлайн-списков («вариант 1 / вариант 2»), нумерованных буллетов, меню с выбором, опций AskUserQuestion — везде

## Достоверность информации

- Ищи в интернете актуальную информацию перед ответом, не полагайся только на внутренние знания
- **Версии — не из памяти.** Любую версию (библиотека, рантайм, CLI, Docker-образ, OS-пакет) проверяй онлайн перед тем, как написать или порекомендовать: библиотеки — через Context7, остальное — WebSearch/WebFetch.

## CLAUDE.md

- Когда инициализируешь CLAUDE.md в проекте (`/init`) — пиши его на английском
- **CLAUDE.md описывает текущее СОСТОЯНИЕ проекта, а не его историю.** Состояние = архитектура, конвенции, актуальные ENV/зависимости, текущая структура, где что искать. НЕ changelog, НЕ release notes, НЕ post-merge отчёты. История релизов уже живёт в `git log` и в `docs/plans/completed/` проекта (или аналоге) — не дублируй её в CLAUDE.md.
- После крупных изменений (новые фичи, архитектурные изменения, обновления зависимостей/стека, структурный рефакторинг, новые конвенции или скрипты) — обновляй CLAUDE.md только если **описание состояния** теперь устарело (например, новый сервис, новая ENV-переменная, обязательная для запуска, удалённый модуль, изменённая архитектура). НЕ дописывай пострелизные отчёты со списками файлов, числом тестов, post-merge фиксами — это раздувает контекст каждой будущей сессии.
- Если замечаешь, что `CLAUDE.md` проекта устарел (упоминает удалённые файлы/зависимости, описывает старую архитектуру, противоречит реальному коду или накопил changelog-записи) — скажи об этом пользователю и предложи обновить/сжать его. Не обновляй молча без подтверждения

## Кодинг

- Перед изменениями сначала ищи по кодовой базе (не предполагай, что что-то не реализовано). Используй Grep/Glob с несколькими запросами, если первый поиск ничего не дал.
- Если нашёл существующую реализацию — расширяй или модифицируй её, не создавай дубликаты
- После реализации или фикса — запускай тесты именно для этого участка кода

## Git

- **`git init` ДО создания любого нетривиального контента, а не после** — даже в «не-софтверных» проектах (заметки, клиентские материалы, ресёрч). Цена нулевая. Нет репо, а собираешься писать значимое → сначала `git init` + коммит текущего состояния.
- **Коммить после каждой осмысленной единицы работы** (документ, итог сессии, законченный черновик, рабочее демо, нетривиальная правка), не только «крупных изменений». Незатреканная работа — невосстановимая; пользователь уже терял её так.
- **Снапшот-коммит перед любой рискованной операцией** (`Write` поверх файла, `rm`, `git checkout --`, `git reset --hard`, массовый рефакторинг, «чистка»): `git add -A && git commit -m "snapshot before <operation>"`. Не спрашивай разрешения — просто делай.
- **Значимый контент незатрекан на входе в проект** — обозначь и предложи baseline-коммит перед новой работой; не редактируй большие объёмы без него.
- После первого коммита проверь, что `.gitignore` исключает: секреты (`.env`, `.env.*.local`, всё кроме `.env.example`), артефакты сборки, `node_modules/`, `__pycache__/`, `.DS_Store`, дебаг-дампы (`.playwright*/`). Заметил секрет в staged — сними из индекса и добавь паттерн.

## Команды терминала

- Всегда выполняй команды терминала сам через инструмент Bash — никогда не проси пользователя запускать команды вручную, если можешь сделать это сам

## Секреты и API-ключи

- Никогда не хардкодь секреты, API-ключи, токены или пароли в коде, скриптах, файлах скиллов, MCP-конфигах или любом другом текстовом документе
- Всегда храни секреты в `.env`-файлах и читай их оттуда в скриптах и коде
- Когда скиллу нужны API-ключи — создай отдельный скрипт (рядом с файлом скилла) и отдельный `.env`-файл; текст скилла должен ссылаться на запуск скрипта, а не содержать ключи
- Никогда не проси пользователя вставлять секреты в чат — вместо этого попроси заполнить `.env`-файл напрямую, чтобы ключи не попадали в историю переписки

## Безопасность .env-файлов

- НИКОГДА не читай, не редактируй, не записывай и иным образом не взаимодействуй с `.env`-файлами, содержащими реальные секреты
- Если нужно создать новый `.env`-файл — пиши его с плейсхолдерами (например `API_KEY=your-api-key-here`) и попроси пользователя заполнить реальные значения самому
- Если нужно показать пользователю, что положить в `.env` — создай или отредактируй `.env.example`, никогда не реальный `.env`
- Если скрипту или коду нужна новая env-переменная — добавь её в `.env.example` с комментарием, затем скажи пользователю скопировать её в свой `.env`
- При отладке проблем, связанных с env-переменными — попроси пользователя проверить значения самому, не читай файл для проверки

## Скиллы

- Пиши описания и тело скиллов на английском
- Выноси скрипты в отдельные файлы рядом с файлом скилла, чтобы не перегружать контекст при загрузке скилла
- Текст скилла должен содержать инструкции по запуску скриптов, а не сами скрипты
- Включай в скилл инструкции по чтению сопутствующих скриптов на случай, когда поведения по умолчанию или конфигурации API недостаточно
- Всегда уточняй куда ставить скилл user scope или project scope, если пользователь не указал

## Планы (скилл /plan)

- Харнесс `/plan` создаёт файлы планов со случайными именами из трёх слов (например `plans/vivid-drifting-kahan.md`). Из-за таких имён директория `plans/` потом нечитаема.
- После записи содержимого плана инструментом `Write` **всегда переименовывай файл** в `YYYY-MM-DD-<kebab-case-summary>.md` через `Bash mv` перед вызовом `ExitPlanMode`. Пример: `plans/2026-04-24-drop-usage-quota-and-llm-quota-used.md`.
- Summary — 3–6 слов в kebab-case, описывающих само изменение (не id тикета и не номер стадии).
